Executive Summary

Goal

Find and connect records that describe the same person, quickly and accurately—even across two different databases.

Benefits

Cleaner data, fewer duplicates, better reporting. Saves review time by auto-accepting only very confident matches (95%+ score).

What's Included

Batch processing (streaming for big jobs), easy exports, friendly GUI, optional GPU boost, and a summary report for audit and KPIs.

Current Workflow & Use Cases

How people use Name_Matcher today—without the jargon.

Deduplicate One Database

Remove repeated people within the same table or schema.

Cross-Database Match

Connect people across two different databases to unify records.

Review by Confidence

Auto-accept results at ≥95%. Keep 85–94.99% for human review. Ignore lower scores.

Typical run pattern
  • Scheduled nightly/weekly batch runs for operations.
  • Ad‑hoc runs in the GUI for analysts.

What the System Must Do (Functional)

  • Match people using three approaches:
    • Method 1: Strict checks using IDs, names, and birthdate.
    • Method 2: A stricter version that also considers middle names.
    • Method 3 (Fuzzy): A “best‑guess” score based on how similar names look/sound. Birthdate must match.
  • Work across one or two databases.
  • Handle huge workloads by streaming; use in‑memory for small jobs.
  • Export results to CSV and/or Excel and create a timestamped summary.
  • Only count fuzzy matches in summaries when the score is 95%+.
  • Let users run and monitor jobs in the GUI; allow automation by scripts.
  • Support checkpoints so long jobs can resume after interruptions.
  • Use the GPU for fuzzy matching if available; otherwise keep going on CPU.

How It Should Behave (Non‑Functional)

Performance

Stream data to keep memory steady; write files efficiently; tune for SSDs.

Security

Use environment variables for credentials, avoid logging secrets, and prevent injection attacks.

Reliability

Clear error messages with fixes. Fall back to CPU when GPU isn’t available.

Usability

Simple GUI with progress, ETA, and helpful status. CLI/automation friendly.

Quality

Unit/integration tests cover matching and exports.

Technical Constraints & Dependencies

  • Built in Rust; connects to MySQL; writes CSV/Excel.
  • GUI uses modern Rust UI libraries; file pickers for convenience.
  • Optional GPU support requires a compatible NVIDIA driver and CUDA runtime.
  • Works on Windows and Linux; uses cross‑platform file paths.

User Interface Requirements

GUI

  • Connect to one or two databases with simple forms.
  • Pick tables, choose method, choose where to save results.
  • Advanced: pool size, batch size, memory threshold, SSD mode, GPU toggle + memory budget.
  • Buttons: Test Connection, Run, Pause/Cancel, Generate .env, Load .env, CUDA Diagnostics.

CLI (Command Line)

  • Script‑friendly options for automation (same core features as GUI).
  • Clear help text; uses exit codes to signal success/failure.

Database Requirements & Compatibility

  • Works with optional fields like middle name and birthdate; safely handles missing values.
  • For best speed, add indexes on name fields and birthdate.
  • Supports an optional UUID in the second table when needed.
The system is careful with missing data so matches remain fair and predictable.

GPU Acceleration (Optional) & Fallbacks

  • When available, the GPU can speed up fuzzy matching.
  • If the GPU isn’t available or fails, the job continues on CPU without crashing.
  • Users can see GPU/driver details in the diagnostics panel.

Configuration & Environment

  • Settings come from a .env file or are set directly in the GUI.
  • Checkpoints (saved next to your output) let you resume long jobs.

Summary Reports & Confidence Threshold

Only fuzzy matches with a score of 95% or higher are counted in the summary statistics. Lower scores are excluded from totals and derived metrics.

Implementation Roadmap & Risks

Near Term

  • Keep the 95% rule consistent everywhere (GUI & CLI).
  • Strengthen diagnostics; allow easy sharing when requesting support.

Future Ideas

  • Support more databases (e.g., PostgreSQL).
  • Job files (JSON/YAML) for repeatable runs; richer audit bundles.

Key Risks

  • GPU/driver issues → mitigate with diagnostics and CPU fallback.
  • Very large runs → use streaming, SSD tuning, and connection pools.
  • Schema differences → validate early; show clear fixes.
  • Sensitive data → keep secrets out of logs; use env vars.
